# Product Discovery Best Practices

Use this reference to keep PM sessions sharp, consistent, and product-led.

## Discovery Principles

- start with the problem, not the UI
- make the actor and desired outcome explicit early
- describe the current workflow before proposing the target workflow
- separate must-learn questions from already-known facts
- prefer a coherent MVP slice over a broad but weak scope
- keep names stable so later phases reuse the same vocabulary

## Good Discovery Questions

- What user pain or business risk exists today?
- Who feels the pain most directly?
- What decision or action must the product help them take?
- What happens before, during, and after the core workflow?
- What states or exceptions would make the experience fail?
- What is the smallest useful version that still changes behavior meaningfully?

## Anti-Patterns

- treating screens as the product instead of the workflow
- collapsing brainstorming into implementation planning too early
- assuming backend behavior from UI without marking it as inference
- overfitting MVP to design completeness instead of user value
- inventing new names for existing platform concepts without reason

## Discovery Quality Bar

- the problem statement is concrete
- actors and outcomes are explicit
- current and target workflows are understandable
- explicit versus inferred behavior is separated
- MVP versus later is defensible
- open questions are visible, not buried
